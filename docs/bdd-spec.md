## 1. 产品概述

**纪念钞设计平台**是一个让普通用户自行设计个性化纸质文创产品的 Web 应用。平台支持多种产品类型——纪念钞、书签、海报等——每种产品有其独立的设计约束和版式模板。用户上传自己喜欢的动漫/游戏/影视角色图片作为"主角"，选择风格、配色等参数，由 AI 生成符合对应产品版面规范的设计图。

**关于打印与交付**：目前，仅"纪念钞"支持提交印刷订单并邮寄实物。其他产品类型（书签、海报等）设计完成后，用户可以导出高清图片自行处理。未来可能扩展其他产品的印刷支持。

**核心价值**：把专业设计师才能做的事，变成任何粉丝都能在几分钟内完成的体验。用户不需要懂设计，只需要有"想让喜欢的角色出现在周边产品上"这个念头。

**商业模式**：在电商平台（淘宝、微信小店等）销售"激活码"，一个激活码兑换一次纪念钞设计+印刷机会。设计过程中的 AI 调用（生图、LLM、网络搜索）由平台承担成本，不计次。非纪念钞产品不需要消耗设计次数即可设计和导出。

**技术路线**：前端 Web 应用 + 后端 API + AI 大脑（LLM 意图理解 + 生图模型）。用户隐私最小化原则：不存储密码（验证码登录）、不存储邮寄地址（浏览器 localStorage）。

## 2. 核心概念速查（Glossary）

在阅读后续场景之前，先理解以下概念：

| 术语 | 含义 |
|------|------|
| **产品类型 (Product Type)** | 用户要设计的产品类别。目前支持三种：**纪念钞**（双面设计，支持印刷邮寄）、**书签**（单面设计，窄长条形，仅导出）、**海报**（单面设计，大幅面，仅导出）。产品类型在创建项目时选择，选定后不可更改。不同类型有不同的设计约束和提示词模板。 |
| **项目 (Project)** | 一次产品设计的完整工作单元。一个项目属于一种产品类型，包含该类型对应的设计稿（纪念钞为正反面各一张，书签和海报为单面）、关联的素材、设计参数。 |
| **设计次数** | 用户持有的"可提交纪念钞印刷订单"的配额。不是创建项目的配额——创建项目不限次数。设计次数通过激活码获得。**仅纪念钞类型的项目在提交印刷时消耗设计次数**；书签和海报类型不需要设计次数，可直接导出。 |
| **激活码** | 在电商平台购买的一串唯一字符，在平台内兑换为设计次数。一个激活码 = 1 次纪念钞设计+印刷机会。 |
| **素材 (Material)** | 用户上传、搜索或从常用库中选用的图片资源。素材是设计的"原材料"——例如角色立绘、Logo、背景图。素材归属于项目，但可以被收藏到用户的常用素材库。 |
| **常用素材库** | 用户级别的素材收藏夹。跨项目复用，类似于"我的贴纸收藏"。用户可以将任意项目中的素材收藏到此库，也可以随时移除。 |
| **AI 大脑** | 平台核心的智能模块，负责两件事：① 从用户自然语言描述中提取结构化设计要素（谁、在哪、什么风格、什么产品）；② 将要素填入对应产品类型的生图提示词模板。对外可能表现为一个对话式机器人或一个后台处理步骤。 |
| **快速模式** | 用户提供素材 + 一段自然语言描述，其余由 AI 大脑自动完成。目标用户：不想纠结细节、只想快速出图的人。 |
| **精细模式** | 用户逐项调节设计分区（风格、配色、主角位置、底纹、文字内容、装饰元素）。每个分区是一个独立的表单区域。不同产品类型暴露的调节分区不同（如纪念钞有"面值"分区，书签和海报没有）。 |
| **正面 / 反面（面数）** | 纪念钞有正反两面。正面通常包含面值、主角人物、发行机构名称、编号；反面通常包含满版防伪底纹、徽章或风景图案。**书签和海报仅单面设计**（不分正反面）。 |
| **生图模型** | 实际生成产品预览图的 AI 图像生成模型（如 Stable Diffusion、Midjourney、DALL·E 等，具体待定）。 |
| **印刷（仅纪念钞）** | 后台管理员将用户提交的纪念钞设计图通过专业设备印刷为实体纪念钞。物理规格（尺寸、纸张、DPI）详见 Feature 10。 |
| **导出（书签/海报）** | 非纪念钞产品设计完成后，用户可下载高清 PNG/JPG 图片，自行去打印店或线上印刷服务商处理。导出不消耗设计次数。 |

## 3. 用户角色

| 角色 | 身份标识 | 权限边界 |
|------|----------|----------|
| **访客 (Guest)** | 无账号 | 浏览平台首页的排行榜和公开项目。无法创建项目、点赞、Fork、提交订单。任何需要身份的操作都将触发引导注册/登录。 |
| **用户 (User)** | 已注册登录的账户 | 创建和管理自己的项目、使用两种模式设计纪念钞、管理素材、提交订单、购买/使用激活码、参与社区互动（点赞、Fork、公开项目）。 |
| **管理员 (Admin)** | 后台账户（本期文档不展开） | 审核已提交的订单、管理印刷流程、更新订单状态。在不暴露用户隐私的前提下与用户协调邮寄事宜。 |

> **注意**：用户角色没有"付费用户"和"免费用户"的区分——用户必须有设计次数（购买激活码）才能提交设计，即所有提交行为都是付费的。浏览和设计草稿阶段不收费。

## 4. Feature 1: 注册与登录

### 4.1 业务背景

**设计决策：为什么不用"邮箱+密码"注册？**

- 密码是敏感信息，需要安全的存储和传输机制（hash、salt、HTTPS 等），增加系统的安全负担。
- 密码会增加用户的心理负担（"又要记一个密码"），降低注册转化率。
- 验证码登录（Magic Link 的思想变体）既安全又简单：用户只需要能收邮件就行。
- 配合第三方登录（微信/Google），绝大部分用户可以在 30 秒内完成注册。

**设计决策：为什么不做手机号注册？**

- 手机号是敏感个人信息。本平台的原则是最小化收集用户数据。邮箱已足够作为唯一标识符和联络方式。
- 如果未来需要短信通知，可以单独让用户可选填写手机号，但不作为注册必选项。

**设计决策：登录态如何维持？**

- 使用 JWT Token 存储在浏览器的 localStorage 或 httpOnly cookie 中。Token 有过期时间（如 7 天），过期后需重新验证。
- 用户关闭浏览器后重新打开，如果 Token 未过期，自动恢复登录态（无需重新验证）。

### 4.2 注册场景

```
Scenario 1.1 — 邮箱注册成功
  前置条件:
    - 用户使用的邮箱 "sakura@example.com" 尚未在系统中注册
    - 用户在浏览器中打开了平台的注册/登录页面

  Given 用户在注册表单中输入邮箱 "sakura@example.com"
   When 用户点击"获取验证码"按钮
   Then 系统向后端发送请求 POST /auth/send-code { email: "sakura@example.com" }
    And 后端生成一个 6 位数字验证码，有效期为 10 分钟
    And 后端通过邮件服务将验证码发送到 "sakura@example.com"
    And 前端显示"验证码已发送到您的邮箱"并展示验证码输入框
    And 前端开始一个 60 秒的倒计时，期间"重新发送"按钮置灰
   When 用户从邮箱中查看验证码，假设为 "482916"
    And 用户在验证码输入框中输入 "482916"
    And 用户点击"验证并注册"
   Then 系统调用 POST /auth/verify-code { email: "sakura@example.com", code: "482916" }
    And 后端校验验证码正确且在有效期内
    And 后端创建用户账户
    And 后端生成一个 6 位随机数字作为用户昵称（如 "用户382491"）
    And 后端返回 JWT Token 和用户基本信息（id, nickname, designQuota: 0）
    And 前端存储 Token，将用户导航至平台首页
    And 首页显示"欢迎，用户382491"


Scenario 1.2 — 邮箱已注册时提示直接登录
  前置条件:
    - 邮箱 "sakura@example.com" 已在此前完成了注册

  Given 用户在注册表单中输入邮箱 "sakura@example.com"
   When 用户点击"获取验证码"
   Then 系统发现该邮箱已注册
    And 系统仍然发送验证码（安全考虑：不泄露邮箱是否已注册）
    And 前端显示标准提示"验证码已发送到您的邮箱"
   When 用户输入正确验证码并提交
   Then 系统返回 JWT Token，直接登录（而非提示"已注册"再让用户切换页面）
    And 用户无感知地完成了"登录"而非"注册"


Scenario 1.3 — 验证码输入错误
  Given 系统向 "sakura@example.com" 发送了验证码 "482916"
   When 用户在验证码输入框中输入 "000000"
    And 用户点击"验证并注册"
   Then 系统提示"验证码错误，请重新输入"
    And 不清空邮箱输入框，不重置倒计时


Scenario 1.4 — 验证码过期
  Given 系统在 10 分钟前向 "sakura@example.com" 发送了验证码 "482916"
   When 用户现在输入 "482916" 并提交
   Then 系统提示"验证码已过期，请重新获取"
    And 验证码输入框清空
    And "重新获取"按钮恢复可用


Scenario 1.5 — 短时间内频繁请求验证码
  Given 用户刚点击过"获取验证码"，60 秒倒计时尚未结束
   When 用户在倒计时期间再次点击"获取验证码"（按钮在 UI 上已置灰，但此处测试绕过）
   Then 系统返回错误"请等待 60 秒后再获取验证码"
    And 不发送新邮件


Scenario 1.6 — 同一邮箱 24 小时内请求验证码超过上限
  Given 邮箱 "sakura@example.com" 在过去 24 小时内已请求过 10 次验证码
   When 用户再次点击"获取验证码"
   Then 系统返回错误"今日请求次数已达上限，请 24 小时后重试"
    And 不发送邮件
  # 目的：防止恶意消耗邮件配额


Scenario 1.7 — 第三方账号注册（微信扫码）
  前置条件:
    - 用户微信未绑定过本平台账号

  Given 用户在注册/登录页面上点击"微信登录"图标
   When 系统跳转到微信 OAuth 授权页面
    And 用户扫码并同意授权
   Then 微信回调本平台，返回授权码
    And 系统用授权码换取微信用户信息（openid, unionid, 昵称, 头像）
    And 系统发现该 openid 未绑定任何账户 → 创建新账户
    And 用户的默认昵称使用微信昵称（而非随机数字）
    And 系统返回 JWT Token，登录成功


Scenario 1.8 — 第三方账号登录（已绑定的微信账号）
  前置条件:
    - 用户此前已通过微信注册过

  Given 用户在平台上点击"微信登录"
   When 用户完成微信授权
   Then 系统发现该 openid 已绑定账户
    And 直接返回 JWT Token，登录成功
```

### 4.3 登录场景

```
Scenario 1.9 — 已注册用户通过验证码登录
  前置条件:
    - 用户已用 "sakura@example.com" 注册过

  Given 用户在登录表单中输入 "sakura@example.com"
   When 用户点击"获取验证码"
   Then 系统发送验证码邮件（与注册流程相同的发送逻辑）
   When 用户输入正确验证码
   Then 系统返回 JWT Token，登录成功
    And 用户看到自己的项目列表（而非空白欢迎页）


Scenario 1.10 — 未注册用户尝试登录
  Given 用户输入一个未注册的邮箱
   When 用户点击"获取验证码"
   Then 系统仍然发送验证码（不暴露邮箱未注册）
    And 前端提示与注册时一致
   When 用户输入验证码
   Then 系统发现该邮箱未注册 → 自动创建账户并登录
    And 用户的体验是"输入邮箱 → 收码 → 进系统"，不需要区分注册和登录
  # 设计意图：注册和登录是同一个流程。用户不需要关心自己是"注册"还是"登录"。
  # 有账户就登录，没账户就自动创建。减少用户决策负担。
```

### 4.4 隐私与注销

```
Scenario 1.11 — 用户注销账号（无公开项目）
  前置条件:
    - 用户有 3 个项目，其中 0 个已公开
    - 所有项目均为草稿或已删除状态

  Given 用户在"设置"页面
   When 用户点击"注销账号"
   Then 系统弹出确认对话框，说明后果：
    - "您的所有项目、素材、设计次数将被永久删除"
    - "此操作不可撤销"
   When 用户输入邮箱地址以确认
    And 用户点击"确认注销"
   Then 系统删除用户账户及所有关联数据
    And 清除该用户的 JWT Token
    And 用户跳转到首页（以访客身份）


Scenario 1.12 — 用户注销账号（有已公开项目）
  前置条件:
    - 用户有 2 个已公开项目

  Given 用户在"设置"页面点击"注销账号"
   When 系统检测到用户有已公开项目
   Then 确认对话框中额外显示选项：
    - "匿名保留公开项目"：项目保留在社区中，作者显示为"匿名用户"
    - "删除所有内容"：公开项目也从社区中移除
   When 用户选择"匿名保留公开项目"并确认
   Then 账户注销，公开项目的作者变为"匿名用户"，其余数据删除
```

### 4.5 边界与异常

| 边界情况 | 处理方式 |
|----------|----------|
| 验证码连续输入错误 5 次 | 该验证码作废，强制重新获取 |
| 邮箱服务不可达 | 前端显示"邮件发送失败，请稍后重试"，不暴露技术细节 |
| 第三方登录授权被拒绝 | 用户回到平台页面，显示"授权已取消"，停留在登录页 |
| JWT Token 过期 | API 返回 401，前端自动清除 Token 并跳转到登录页，保留用户当前浏览的 URL 作为重定向目标 |
| 用户更换浏览器 / 清除缓存 | 需要重新登录。由于只有验证码登录，用户不担心"忘记密码" |

## 5. Feature 2: 项目管理（含产品类型选择）

### 5.1 业务背景

"项目"是用户一次产品设计的基本工作单元。它不是"一个文件"，而是一个包含以下数据的容器：

- 项目元数据：名称、产品类型（创建时选定，不可更改）、创建时间、最后修改时间、当前状态
- 与设计相关的数据：快速模式 / 精细模式的参数、AI 大脑生成的提示词历史
- 设计稿：纪念钞为正反面各一张图，书签和海报为单面一张图
- 素材关联：该项目引用了哪些素材

**设计决策：为什么用"项目"而不是直接让用户操作图片？**

产品设计不是一个原子操作。用户可能需要多次迭代（生成 → 不满意 → 修改描述 → 重新生成），换素材，换模式。一个项目承载这个迭代过程的所有上下文。此外，"项目"天然适合社区功能——Fork 他人的项目意味着复制整个设计上下文，而不只是一张图。

**设计决策：为什么产品类型在创建时选定且不可更改？**

纪念钞（160mm×80mm 横版双面）、书签（50mm×150mm 竖版单面）、海报（A4/A3 单面）的物理比例和版面约束完全不同。如果在设计中途切换产品类型，之前生成的设计稿和配置参数将全部失效。与其让用户困惑，不如一开始就明确选择。

**状态机说明：**

纪念钞类型项目：
```
[设计稿] ──提交打印──▶ [已提交] ──管理员处理──▶ [已印刷] ──用户公开──▶ [已公开]
   │                      │                        │                      │
   └── 可编辑             └── 不可编辑              └── 可选择公开          └── 可撤回公开
                           等待管理员审核           印刷完成                （回到"已印刷"）
```

书签/海报类型项目：
```
[设计稿] ──导出──▶ [已完成] ──用户公开──▶ [已公开]
   │                 │                      │
   └── 可编辑        └── 可选择公开          └── 可撤回公开
```

- **设计稿**：用户正在编辑中，可随时修改任何内容
- **已提交**（仅纪念钞）：用户已确认并提交印刷，扣减了设计次数，等待管理员处理。此状态不可再编辑
- **已印刷**（仅纪念钞）：管理员完成印刷，项目中多了印刷成品的相关信息
- **已完成**（书签/海报）：用户完成设计并导出
- **已公开**：用户选择将设计分享到社区排行榜。公开后不可再撤回为草稿（但可取消公开回到上一状态）

## 6. Feature 3: 素材管理

### 6.1 业务背景

素材是用户设计的原材料。常见素材类型举例：

- 动漫角色的立绘/截图（如《火影忍者》鸣人、《鬼灭之刃》炭治郎）
- 品牌 Logo（如二次元周边店"黄师傅"的 Logo）
- 背景图片（如富士山、竹林）
- 装饰元素（如边框、徽章矢量图）

素材来源有三个渠道，对应素材面板中的三个 Tab：

| Tab | 素材来源 | 存储归属 | 生命周期 |
|-----|---------|---------|---------|
| 本地上传 | 用户从电脑/手机上传 | 归属于当前项目，上传的文件存储在云端 | 项目删除时一并删除 |
| 网络搜索 | 通过内置搜索引擎从互联网检索 | 用户选中后复制到项目素材库 | 与项目同生命周期 |
| 常用素材 | 用户收藏的素材 | 归属于用户账户，跨项目可复用 | 用户手动取消收藏前一直保留 |

**设计决策：为什么要有"常用素材库"？**

如果用户的每张纪念钞都用同一个品牌 Logo（比如"黄师傅 Logo"），每次新建项目都重新上传会非常烦人。"常用素材库"相当于一个跨项目的素材收藏夹，解决了素材复用的问题。

## 7. Feature 4: 快速模式设计

### 7.1 业务背景

快速模式是整个产品体验的核心——它将"设计纪念钞"这个行为的门槛降到最低。用户的输入只有两个：素材 + 一段自然语言描述。其余的事情由 AI 大脑完成。

**关于正反面设计的说明：**

一张纪念钞有正面和反面两面。正面通常放置面值、主角图像、发行机构等"门面"信息；反面通常放置防伪底纹、风景图案、徽章等"支撑"元素。用户需要分别设计两面——正面和反面在编辑器中是两个独立的 Tab。

两面可以分别使用不同模式。例如：正面用快速模式快速出图，反面用精细模式微调底纹。这给了用户灵活性。

**关于生图过程的说明：**

生图模型（如 Stable Diffusion）的调用是耗时的——可能需要 5~30 秒甚至更长。在这期间：
- 前端显示生成进度（不是精确百分比，而是阶段提示："正在理解你的描述…" → "正在生成纪念钞正面…" → "正在润色细节…"）
- 用户可以切换到其他 Tab 或做别的事（后台生成，但页面上有状态提示）
- 生成完成后前端收到通知（WebSocket 或轮询），自动展示预览图

**设计决策：为什么保留历史生成记录？**

生图不是确定性的——同一个描述生成两次，结果往往不同。保留历史记录让用户可以回溯之前某一次效果更好的生成结果，而不需要每次都从头开始。

## 8. Feature 5: 精细模式设计

### 8.1 业务背景

精细模式面向的是对纪念钞有明确想象的用户——他知道想要什么风格、什么配色、主角放在哪个位置、边框长什么样。精细模式把快速模式中"藏在 AI 大脑背后"的参数全部暴露出来，让用户逐项调控。

精细模式的本质是：**将生图提示词拆解为结构化的参数表单**。用户填写表单 → 系统组装提示词 → 生图。用户也可以查看和手动编辑组装的提示词（高级用户）。

精细模式和快速模式共享同一个生图历史面板，但精细模式下每次生成会顺手保存当前的参数快照，方便回溯。

---

### 8.2 精细模式下的设计分区

以下是精细模式的设计分区及其含义。注意这些分区不是 UI 的最终布局，而是需要被控制的逻辑维度：

| 分区 | 说明 | 参数类型 |
|------|------|----------|
| **整体风格** | 纪念钞的整体视觉风格 | 预设选项（复古 / 现代 / 赛博朋克 / 日式浮世绘 / 新古典主义 / …）+ "自定义描述"输入框 |
| **配色方案** | 纪念钞的色调 | 预设色板（可点击选择）+ 自定义 hex 色值输入（主色、辅色、强调色） |
| **面值** | 纪念钞的面值 | 数字输入（正整数或小数） + 货币单位输入（如"日元"、"两"、"贝利"、"积分"，默认"元"）|
| **主角元素** | 纪念钞上最主要的人物/角色/Logo | 从项目素材库中选择 1 个素材，设置其位置（下拉：正中 / 左侧 / 右侧 / 自定义），设置大小（滑块：小/中/大），设置效果（下拉：无 / 雕刻版画效果 / 水印效果） |
| **背景底纹** | 防伪扭索纹的样式 | 纹样选择器（预设 5-10 种扭索纹样式），密度滑块（疏 / 中 / 密），防伪线位置（下拉：底部 / 左侧 / 环绕 / 无） |
| **文字内容** | 纪念钞上的文字信息 | 发行机构名称（如"二次元纪念钞发行局"）、系列名称（如"火影忍者系列"）、自定义文字（可选，展示在纪念钞上某处） |
| **编号规则** | 纪念钞的序列号格式 | 前缀字符（如 "NC"）、数字位数（如 6 位 = 000001~999999）、是否随机生成 |
| **装饰元素** | 边框纹样、四角装饰等 | 边框样式（预设 5-10 种：简约线条 / 复杂古典纹 / 齿轮工业风 / …），四角装饰（开启/关闭），是否添加水印区域 |

```
Scenario 5.1 — 精细模式正面设计（完整流程）
  前置条件:
    - 用户有项目，当前处于"正面"Tab
    - 用户已在素材库中添加了素材 naruto-hero.png

  Given 用户在编辑页顶部点击模式切换按钮，从"快速模式"切换到"精细模式"
   When 模式切换完成
   Then 编辑页左侧的输入区变为设计分区面板，以手风琴或分组卡片形式展示
    And 当前显示的分区为上述 8 个分区的表单控件

  # 用户逐一调整参数
   When 用户在"整体风格"中选择"日式浮世绘"，并在"自定义描述"中补充"葛饰北斋风格的海浪"
    And 用户在"配色方案"中选择预设色板"浮世绘蓝白"，手动将强调色改为 #C41E3A（朱红）
    And 用户在"面值"中输入数字 5000，单位为"两"
    And 用户在"主角元素"中从素材库选择 naruto-hero.png，位置选择"右侧"，大小选择"大"，效果选择"雕刻版画效果"
    And 用户在"背景底纹"中选择"和风波浪纹"，密度"中"，防伪线位置"底部"
    And 用户在"文字内容"中输入发行机构"木叶隐村纪念钞发行局"，系列名称"第七班系列"
    And 用户在"编号规则"中前缀设为"NC"，位数保持默认 8 位，勾选"随机生成"
    And 用户在"装饰元素"中选择边框样式"和风几何纹"，开启四角装饰

  # 用户点击预览
   When 用户点击"生成预览"按钮
   Then 系统将用户的所有参数传给 AI 大脑
    And AI 大脑将这些参数组装为完整的生图提示词
    And 系统调用生图模型生成预览图
    And 预览图展示在画布中
    And 画布旁显示当前参数快照（只读摘要："风格：日式浮世绘 | 配色：浮世绘蓝白 | 面值：5000两 | …"）


Scenario 5.2 — 精细模式切换到快速模式（参数保留）
  前置条件:
    - 用户在精细模式下调整了许多参数并生成了预览

  Given 用户当前在精细模式
   When 用户将模式切换回"快速模式"
   Then 系统保留精细模式下的参数配置（不删除）
    And 用户可在快速模式中用自然语言描述
    And 快速模式的描述会覆盖精细模式的部分参数（描述中提到的部分）
    And 如果用户再次切回精细模式，之前的精细化参数仍然保留
  # 用户可以反复横跳，两边不丢数据


Scenario 5.3 — 精细模式 - 反面设计独立配置
  Given 用户在正面使用精细模式配置了"赛博朋克+鸣人+面值10000"
   When 用户切换到"反面"Tab
   Then 反面的精细模式分区展示反面特有的默认值：
    - 整体风格：默认"经典防伪"
    - 配色：默认蓝灰色系
    - 无面值分区（反面通常不展示面值）
    - 无主角元素分区（反面通常不放大主角，换成"徽章/风景图案"选择器）
    - 背景底纹：默认"扭索纹-经典"，密度"密"
    - 有水印区域分区
    And 正面的配置不影响反面，反面是独立的一套参数


Scenario 5.4 — 精细模式下某些分区留空
  前置条件:
    - 用户在精细模式下只填写了风格、配色、面值、主角四个分区
    - 背景底纹、文字内容、编号规则、装饰元素为空

  Given 用户点击"生成预览"
   When AI 大脑组装提示词时遇到空分区
   Then 这些空分区使用预设的默认值（不跳过）
    - 背景底纹 → 默认"扭索纹-经典"
    - 文字内容 → 默认"纪念钞发行局"
    - 编号规则 → 默认随机 8 位数字
    - 装饰元素 → 默认"简约边框"
   And 预览图仍然完整生成（不会因为用户没填装饰元素就缺边框）
```

## 9. Feature 6: AI 大脑（意图理解与提示词组装）

### 9.1 业务背景

"AI 大脑"是本平台的"翻译官"——它负责把用户的自然语言描述或表单参数翻译成生图模型能理解的高质量提示词。这是产品体验的核心差异化能力，因为它决定了"用户随便说一句话，能不能出一张像样的纪念钞"。

**AI 大脑由两部分组成：**

1. **意图提取器**：输入 = 用户的自然语言描述 + 已选素材信息；输出 = 结构化设计要素 JSON。本质上是一个 LLM 调用，使用精心设计的 system prompt 来约束输出格式。
2. **提示词组装器**：输入 = 结构化设计要素 JSON + 纪念钞正/反面模板；输出 = 完整的生图提示词。可以是规则引擎（要素填入模板的预留槽位），也可以再次使用 LLM 进行润色组装。

**技术选型待定**：可以自研一个简单的 LLM agent（如基于 GPT-4 或 Claude 的 function calling），也可以接入已有的 agent 框架（如 pi-agent）。关键在于 prompt engineering——模板的质量直接决定生成效果。

**费用考量**：AI 大脑的 LLM 调用成本 + 生图模型的调用成本，都由平台承担，不计入用户的设计次数。用户支付的是"印刷 + 邮寄"的物理成本，设计过程中的 AI 费用是获客成本的一部分。

### 9.2 意图提取

```
Scenario 6.1 — AI 从自然语言中提取设计关键要素（正面，信息完整）
  前置条件:
    - AI 大脑的意图提取器已部署

  Given 用户输入如下描述：
    "把鸣人放在中间偏右，背景是木叶村的夕阳，面值 5000 两，风格偏复古浮世绘，
     整体暖色调，边框用金色传统纹样，加上木叶村的标志"
    And 用户已选素材：
    - naruto-stand.png（鸣人立绘）
    - konoha-logo.png（木叶村标志）

   When 系统将上述信息发送给 AI 大脑的意图提取器
   Then AI 提取出以下结构化 JSON（示意）：
    {
      "mainSubject": {
        "materialRef": "naruto-stand.png",
        "position": "slightly right of center",
        "size": "default"
      },
      "background": "木叶村夕阳",
      "denomination": {
        "value": 5000,
        "unit": "两"
      },
      "style": "复古浮世绘 / vintage ukiyo-e",
      "colorScheme": "暖色调 / warm tones",
      "border": "金色传统纹样 / gold traditional pattern",
      "additionalElements": ["木叶村标志"],
      "extraMaterials": ["konoha-logo.png"]
    }
    And 提取耗时在 3 秒以内


Scenario 6.2 — AI 从自然语言中提取反面设计要素
  Given 用户输入反面描述：
    "经典防伪扭索底纹，中间放一个大大的木叶徽章，底部一条银色防伪线，
     背景用水印效果的树叶图案，整体青灰色调，严肃正式的感觉"
   When AI 大脑处理该描述
   Then 提取出反面特有的结构化要素：
    {
      "guilloche": "经典扭索纹 / classic guilloche",
      "centralEmblem": {
        "description": "木叶徽章",
        "size": "large"
      },
      "securityThread": {
        "position": "bottom",
        "color": "银色 / silver"
      },
      "watermark": "水印效果树叶图案 / watermark leaf pattern",
      "colorScheme": "青灰色调 / blue-grey tones",
      "mood": "严肃正式 / serious and formal"
    }


Scenario 6.3 — AI 处理描述缺乏关键信息时的追问策略
  前置条件:
    - 用户在快速模式下的描述不包含面值、风格等关键信息

  Given 用户输入："帮我做一个好看的纪念钞"（过于模糊）
   When AI 大脑尝试提取要素
   Then AI 无法可靠地填充多个关键槽位（风格、配色、背景均未提及）
    And 系统不生成图像，而是在画布区域展示 AI 的追问：
      "我注意到了你的素材，还有几个关键信息需要确认：
       ① 你希望是什么风格？比如赛博朋克、复古日式、现代简约…
       ② 面值设定为多少？比如 1000、10000
       ③ 有没有特别想要的色调？比如暖色、冷色、黑金…"
    And 追问以对话气泡的形式展示（像一个聊天机器人在提问）
    And 用户可以在下方的回复框中逐一回答
    And 回答会累积到描述文本中，用户再次点击"生成"

  # 设计意图：不要让 AI 自己瞎猜然后生成一个用户不满意的结果。
  # 追问的成本（多一轮 LLM 调用）远小于生成一张不满意的图（生图成本 + 用户失望）。


Scenario 6.4 — AI 追问只进行一次
  前置条件:
    - AI 已经追问过一次，用户补充了信息但可能仍不完整

  Given 用户被追问后回复："风格赛博朋克，面值 1000"（仍未提及配色和背景）
   When 系统将补充信息合并到原始描述中
    And 用户再次点击"生成"
   Then AI 不再追问第二次
    And 对于仍然缺失的要素（如配色、背景），使用默认值补齐
    And 正常生成预览图
  # 追问一次就够了，追两次会让用户烦躁。宁可给个默认值出图，用户不满意自会修改。
```

### 9.3 提示词组装（按产品类型选择模板）

```
Scenario 6.5 — AI 根据产品类型选择不同的提示词模板
  前置条件:
    - AI 已提取出结构化设计要素
    - 当前项目产品类型已知

  Given 系统调用提示词组装器，传入提取的要素 JSON 和产品类型标识
   When 产品类型为"纪念钞"

Scenario 6.6 — AI 将提取的要素填入纪念钞正面提示词模板
  前置条件:
    - AI 已提取出 Scenario 6.1 中的结构化要素
    - 当前项目为纪念钞类型

   Then 生成的最终提示词大致为（以英文 SD prompt 形式）：

    And 提示词中明确保留了纪念钞的固定要素（编号预留区、防伪标记、高分辨率等）
    And 用户描述中没有提到的要素（如编号位置）由模板默认值填充


Scenario 6.7 — AI 将提取的要素填入书签提示词模板（对比）
  前置条件:
    - AI 提取出与 Scenario 6.6 相同的用户意图（风格、配色、主角）
```

```
Scenario 6.8 — 生图提示词的固定约束不被用户输入覆盖
```

### 9.5 AI 大脑的异常处理

```
Scenario 6.10 — LLM 返回的 JSON 格式不符合预期
  Given 用户输入了一段正常描述
   When AI 大脑调用 LLM 提取要素
    But LLM 返回的 JSON 缺少必填字段（如 denomination 字段缺失）
   Then 组装器检测到缺少必填字段
    And 如果当前产品类型是纪念钞：面值使用默认值 100 + "元"
    And 如果当前产品类型是书签/海报：跳过 denomination 处理（该类型不需要面值）
    And 仍然正常生成提示词（不阻塞流程）
    And 后端记录一条 warning 日志："AI 提取要素缺失字段 [denomination]，使用默认值"


Scenario 6.11 — LLM 服务不可用
  Given AI 大脑依赖的 LLM API 返回 5xx 或超时
   When 系统尝试提取意图
   Then 系统重试 1 次（等待 2 秒后重试）
    And 如果重试仍然失败：
    - 如果是快速模式：前端显示"AI 服务暂时繁忙，请稍后重试或切换到精细模式手动配置"
    - 如果是精细模式：直接使用参数填充模板（精细模式的参数已经结构化，不需要 LLM 提取意图）
  # 精细模式不依赖 LLM ——这是一个重要的架构冗余设计。
```

## 10. Feature 7: 项目提交与订单（纪念钞）+ 非打印产品导出（书签/海报）

### 10.1 业务背景

当用户对设计满意后，接下来的操作取决于产品类型：

- **纪念钞**：提交印刷订单 → 扣减设计次数 → 填写邮寄地址 → 管理员印刷邮寄
- **书签 / 海报**：导出高清图片 → 项目标记为"已完成" → 用户自行处理图片

**核心差异表：**

| 维度 | 纪念钞 | 书签 | 海报 |
|------|--------|------|------|
| 提交/导出按钮文案 | "提交印刷" | "导出图片" | "导出图片" |
| 消耗设计次数 | 是（扣 1 次） | 否 | 否 |
| 需要填写邮寄地址 | 是 | 否 | 否 |
| 生成订单 | 是 | 否 | 否 |
| 项目状态变更 | 设计稿 → 已提交 | 设计稿 → 已完成 | 设计稿 → 已完成 |
| 是否可重新编辑 | 提交后不可 | 可（点击重新编辑回到设计稿） | 可 |

**设计决策：为什么书签/海报不直接支持印刷？**

印刷需要建立供应链（印刷厂对接、质量控制、邮寄流程），首期先跑通纪念钞这一条印刷链路。书签和海报作为免费功能，可以：
1. 降低用户首次体验的心理门槛（不需要买激活码就能玩）
2. 作为获客渠道（用户做了免费书签，分享到社区，吸引更多人）
3. 未来可以接入更多印刷品类（先让用户用起来，有需求再接入供应链）

### 10.2 纪念钞：提交印刷

```
Scenario 7.1 — 纪念钞用户提交印刷（完整流程）

    - 设计参数摘要（风格、面值等，如果有）
    - 提示："提交后将扣减 1 次设计次数（当前剩余：2 次），且设计不可再修改"
    - "确认提交"按钮和"再改改"按钮
   When 用户点击"确认提交"
   Then 系统执行提交前检查：
    
    Check 1 — 正反面完整性（仅纪念钞需要）
    项目必须有正面的预览图 AND 反面的预览图
    → 不通过时提示："请完成纪念钞正面设计"或"请完成纪念钞反面设计"

    Check 2 — 设计次数余额
    用户当前设计次数余额必须 >= 1

    Check 3 — 邮寄地址
    浏览器 localStorage 中是否有缓存的邮寄地址？

    And 所有检查通过后：
    - 系统将项目状态从"设计稿"更新为"已提交"
    - 扣减 1 次设计次数（用户余额从 2 变为 1）
    - 创建订单记录，包含：项目 ID、最终导出图（正反面高分辨率 PNG）、邮寄地址（从浏览器缓存读取）、提交时间
    - 前端显示成功页："设计已提交！管理员将在 1-3 个工作日内处理您的订单"


Scenario 7.2 — 纪念钞提交时正反面不完整
  前置条件:
    - 用户只在正面生成了预览图，反面 Tab 从未操作过

  Given 用户点击"提交印刷"
   When 系统检查到反面无预览图
   Then 对话框关闭
    And 编辑页自动切换到"反面"Tab
    And 顶部显示醒目的提示条（橙色/黄色）：
    "请先完成纪念钞反面的设计。即使是简单的防伪底纹，也需要确认反面设计后才能提交。"
    And "生成"按钮高亮闪烁引导用户操作


Scenario 7.3 — 纪念钞提交时设计次数不足
  前置条件:
    - 用户设计次数余额为 0

  Given 用户点击"提交印刷"
   When 系统检查到设计次数余额为 0
   Then 弹出激活码引导对话框：
    - "当前没有可用设计次数"
    - "您的设计稿已保存，购买激活码后可直接提交，无需重新设计"
    - "如何获取激活码？" 按钮 → 跳转激活码说明页
    - "关闭" 按钮 → 返回编辑页
    And 设计次数余额仍为 0，项目状态仍为"设计稿"


Scenario 7.4 — 纪念钞提交时设计不完整（非反面问题，比如素材缺失）
  Given 用户正面用了素材 naruto.png，但该素材在上传后被从项目中移除了
```

## 11. Feature 8: 激活码（付费机制）

### 11.1 业务背景

**商业模式**：用户在外部电商平台（淘宝、微信小店等）购买激活码 → 在本平台输入激活码兑换设计次数 → 使用设计次数提交纪念钞印刷订单。**激活码仅用于纪念钞的印刷提交**；书签和海报的设计和导出完全免费，不需要激活码。

**设计决策：为什么只对纪念钞收费？**

- 纪念钞有物理印刷和邮寄成本，需要用激活码覆盖。
- 书签和海报仅产生 AI 调用成本（生图 + LLM），边际成本低，可作为免费获客手段。
- 用户先免费体验书签/海报设计 → 对平台产生信任和兴趣 → 购买激活码设计纪念钞（自然的转化漏斗）。
- 如果未来接入书签/海报的印刷供应链，可以相应地推出对应产品的激活码或直接在线支付。

**为什么选择激活码模式而不是直接在线支付？**

1. 避免平台内嵌支付系统（无需对接支付接口、无需 PCI-DSS 合规、无需处理退款逻辑）
2. 利用已有电商平台的信誉和流量（用户更愿意在淘宝下单而不是在一个小众网站上输信用卡）
3. 激活码可以作为礼物赠送（"送你一个纪念钞设计机会"比"转你 50 块钱"更有趣）
4. 激活码 = 天然的反欺诈屏障（每个码只能使用一次，计算消耗量简单明了）

**激活码的生成与管理**：

- 激活码格式：预生成一批唯一字符串（如 "NC-A1B2-C3D4"），管理员在后台生成后，手动上架到电商平台。
- 每个激活码的状态：未售出（初始）→ 已售出（电商平台发货后）→ 已使用（用户在平台兑换后）。
- 如果激活码一直未被使用，永久有效（不过期）。

**设计次数消费规则**：

- 1 个激活码 = 1 次设计次数 = 1 次提交机会
- 设计次数余额是用户账户级别的属性，不是项目级别的
- 用户在项目中多次调用生图模型、LLM、网络搜索，均不额外扣减次数
- 仅在"提交设计"时扣减 1 次
- 提交后如果管理员因技术原因无法印刷（如设计图分辨率不足），可退款/补次数（管理员手动操作）

```
Scenario 8.5 — 设计过程中不限制 AI 调用（核心计费规则验证）
```

## 12. Feature 9: 社区与排行榜

### 12.1 业务背景

社区是本平台的增长引擎。用户看到别人有趣的纪念钞设计，会产生"我也想做一张"的冲动。排行榜提供发现性，Fork 机制降低创作门槛（"别人的设计不错，拿来改改就是我的"）。

**设计决策：为什么允许"直接打印"他人的设计？**

如果用户只是喜欢某张纪念钞但不想自己设计，直接打印是最高效的路径。同时这对原创作者也是一种认可——"我的设计好到别人愿意直接拿钱打印"。目前的设计中，直接打印不会给原创作者分润，但这是一个未来的可能性。

**隐私考量：公开项目暴露什么信息？**

公开项目展示：
- 纪念钞正反面预览图
- 作者昵称（不是邮箱，不是真实姓名）
- 设计参数摘要（面值、风格……）
- 点赞数、Fork 数、浏览量

公开项目**不**暴露：
- 作者邮箱
- 邮寄地址（显然不会在公开内容中）
- 项目中的原始素材（用户上传的角色图等可能有版权风险，不在公开页展示原图，只展示最终生成的纪念钞成品图）

```
    And 项目 B 公开了 2 天，有 50 点赞、10 Fork、500 浏览量

    And 项目 A 的时间衰减因子 = 1 / (1 + 30/30) = 1/2 = 0.5 （半衰期 30 天）
    And 项目 A 的最终热度 = 1400 * 0.5 = 700

    And 项目 B 的原始热度 = 50*3 + 10*5 + 500*1 = 150 + 50 + 500 = 700
    And 项目 B 的时间衰减因子 = 1 / (1 + 2/30) ≈ 0.9375
    And 项目 B 的最终热度 = 700 * 0.9375 ≈ 656

    And 在"热门"排序中，项目 A 排在项目 B 前面（700 > 656）
  # 这是示意算法，实际参数可以根据运营数据调整。


Scenario 9.6 — 用户对公开项目点赞
  前置条件:
    - 用户已登录
    - 排行榜中有一个项目，当前点赞数为 10

  Given 用户将鼠标移到项目卡片上，点击心形按钮
   When 系统收到点赞请求
   Then 检查该用户是否已对该项目点过赞
    And 如果未点过赞 → 点赞数 +1，心形变红（实心）
    And 如果已点过赞 → 点赞数 -1，心形恢复空心（切换式点赞/取赞）
   When 用户刷新页面
   Then 该用户的点赞状态被记住（红心/空心）


Scenario 9.7 — 访客试图点赞
  Given 用户未登录
   When 用户点击项目卡片上的心形按钮
   Then 心形短暂高亮后无变化
    And 弹出登录引导："登录后即可为喜欢的作品点赞"
    And 用户可选择关闭弹窗或跳转登录页
```

### 12.4 Fork 与直接打印

```
Scenario 9.8 — 用户 Fork 他人的公开项目
  前置条件:
    - 用户已登录
    - 用户正在浏览一个公开项目（项目 X，作者：用户 A）

  Given 用户在项目 X 的详情页
   When 用户点击"拿来吧你"（Fork）按钮
   Then 系统弹出确认对话框：
    - "将复制该项目到你的项目列表，你可以在原设计基础上自由修改"
    - "Fork 后你将拥有一个独立的副本，原作者的版本不受影响"
    - "确认 Fork" / "取消"
   When 用户点击"确认 Fork"
   Then 系统创建一个新项目，数据如下：
    - 名称：原项目名称后加" (Fork)"后缀，如"木叶村纪念钞 (Fork)"
    - 状态：设计稿
    - 正反面预览图：从项目 X 复制（作为起点）
    - 设计参数：从项目 X 复制
    - 素材：**不复制**原项目的素材（因为素材可能受版权限制）
    - 来源标记：Fork 自项目 X（项目详情中展示"来源：用户A的'木叶村纪念钞'"）
    - 所属用户：当前用户
   Then 导航到新项目的编辑页
    And 因为素材未复制，页面上提示：
    "该项目 Fork 自 '木叶村纪念钞'。原始素材未复制，你可以自己上传素材替换。"
    And 原项目 X 的 Fork 数 +1


Scenario 9.9 — 用户 Fork 后直接修改并提交
  前置条件:
    - 用户刚 Fork 了项目 X

  Given 用户在被 Fork 的项目编辑页中
    And 用户上传了自己的素材（替代原项目的素材）
    And 用户基于原设计微调了描述或精细模式参数
    And 用户点击"生成"获得新的预览图
   When 用户最终点击"提交设计"
   Then 消耗用户自己的 1 次设计次数（不是原作者的次数）
    And 提交流程与正常项目完全相同
    And 项目详情中仍保留 Fork 来源信息


Scenario 9.10 — 从他人项目直接打印（不进入编辑器）
  前置条件:
    - 用户已登录
    - 用户设计次数余额 >= 1

  Given 用户在项目 X 的详情页
   When 用户点击"直接打印"按钮
   Then 系统弹出确认对话框：
    - "将使用该设计的正反面直接提交订单，不进入编辑器"
    - "确认后将扣减 1 次设计次数"
    - "确认打印" / "取消"
   When 用户点击"确认打印"
   Then 系统检查浏览器中是否有缓存的邮寄地址
    - 如果有：自动填充地址，用户确认后提交
    - 如果没有：引导用户填写地址后提交
    And 系统扣减 1 次设计次数
    And 订单创建，与原项目 X 关联（统计维度：原项目 X 的直接打印次数 +1）
    And 用户自己的项目中不新增项目（因为没有进入编辑器）


Scenario 9.11 — Fork 链与溯源
  前置条件:
    - 项目 A（原创，作者 α）→ 被 Fork 为项目 B（作者 β）
    - 项目 B 被 Fork 为项目 C（作者 γ）

  Given 用户在项目 C 的详情页
   Then 项目详情中展示溯源链：
    "当前项目 Fork 自 → 项目 B（作者 β）→ 原始项目 项目 A（作者 α）"
    And 每条溯源链接可点击跳转到对应项目
  # 这是对原创作者的尊重和社区透明度的体现。
```

### 12.5 访客浏览与转化

```
Scenario 9.12 — 访客在排行榜中被吸引后注册
  前置条件:
    - 访客（未登录）正在浏览首页排行榜

  Given 访客看到一个喜欢的纪念钞，点击卡片进入详情页
    And 详情页展示了完整的设计信息，包括面值、风格描述、正反面图
   When 访客点击"我也要做一张"或"Fork 此设计"
   Then 系统弹出登录引导：
    - "登录后即可创建你自己的纪念钞！"
    - 登录/注册表单（邮箱 + 验证码）
   When 访客完成注册/登录
   Then 系统回到该项目的详情页（而非跳转到首页）
    And 用户之前点击的按钮（"Fork 此设计"或"我也要做一张"）自动触发
    And 用户体验是"点击 → 登录 → 直接进入创建"，而不是"点击 → 登录 → 回到首页 → 重新找"
  # 转化路径的连续性至关重要——不要让用户在登录后丢失了他们的初始意图。
```

## 13. Feature 10: 产品类型与设计约束

### 13.1 业务背景

不同产品类型有不同的物理形态、版面比例和设计约束。这些约束在 AI 大脑的提示词模板中体现，也影响了编辑器的 UI 结构（是否有正反面切换、有哪些设计分区）。

**三种产品类型对比总览：**

| 维度 | 纪念钞 | 书签 | 海报 |
|------|--------|------|------|
| **面数** | 双面（正面 + 反面） | 单面 | 单面 |
| **物理尺寸（暂定）** | 160mm × 80mm（横版） | 50mm × 150mm（竖版） | A4 (210mm × 297mm) 或 A3 |
| **宽高比** | 2:1 | 1:3 | ≈1:1.414 (A4) |
| **是否可打印** | ✅ 提交印刷订单 | ❌ 仅导出图片 | ❌ 仅导出图片 |
| **是否消耗设计次数** | ✅ 提交时扣 1 次 | ❌ | ❌ |
| **精细模式特殊分区** | 面值、编号规则、发行机构 | 无面值、无编号 | 无面值、无编号 |
| **提示词模板关键词** | banknote, guilloche, security, denomination | bookmark, narrow tall, elegant, ribbon | poster, large format, bold, display |
| **生成分辨率** | ≥ 1890×945 (300 DPI) | ≥ 591×1772 (300 DPI) | ≥ 2480×3508 (A4 @ 300 DPI) |
| **导出格式** | 提交时自动导出 PNG | 用户可选择 PNG/JPG | 用户可选择 PNG/JPG |

---

### 13.2 纪念钞设计约束

```
Scenario 10.1 — 纪念钞正面必须包含的要素
  Given 用户在快速模式或精细模式下设计纪念钞正面
   When AI 大脑组装生图提示词
   Then 提示词中始终包含以下要素的描述：

    (1) 面值
    - 数字 + 货币单位，以大字体醒目显示
    - 通常位于纪念钞上部或中部（由模板决定）
    - 精细模式下用户可设定具体的数字和单位
    - 快速模式下 AI 从用户描述中提取，提取不到则默认使用 100 + "元"

    (2) 编号预留区
    - 一个视觉上明确的空白区域（或半透明文字提示"NO. ________"）
    - 通常位于纪念钞左上角或右上角
    - 最终印刷时由管理员在此区域打印序列号
    - 用户不能通过描述去掉此区域（见 Scenario 6.6）

    (3) 发行机构名称
    - 精细模式：用户可在"文字内容"分区自定义（如"木叶隐村纪念钞发行局"）
    - 快速模式：AI 从用户描述中提取
    - 默认值："纪念钞发行局"（系统预设）

    (4) 防伪标记区域
    - 水印位置（通常为圆形或椭圆形区域，位于空白处）
    - 防伪线（一条横穿纪念钞的细线，带有微缩文字或全息效果描述）
    - 扭索纹边框（Guilloche pattern border，细密的花纹边框）


Scenario 10.2 — 纪念钞反面必须包含的要素
  Given 用户在快速模式或精细模式下设计纪念钞反面
   When AI 大脑组装生图提示词
   Then 提示词中始终包含以下要素的描述：

    (1) 满版防伪底纹
    - 整个反面的背景由细密的扭索纹（guilloche）覆盖
    - 纹样从预设的 5-10 种中选择（精细模式）或由 AI 根据风格自动选择（快速模式）
    - 密度：精细模式可调（疏/中/密），快速模式默认"密"

    (2) 徽章或风景图案
    - 位于反面中央的一个视觉焦点元素
    - 可以是圆形徽章、盾形纹章、风景插图（如富士山剪影）
    - 从素材库中选择或由 AI 从用户描述中理解

    (3) 防伪线 / 水印
    - 一条贯穿或环绕的防伪线（金属色效果）
    - 一个或多个水印区域（半透明图案叠加在底纹上）


Scenario 10.3 — 纪念钞分辨率与印刷规格
  Given 生图模型生成了纪念钞预览图
   Then 预览图在网页上显示时可以做适当压缩（如显示为 800px 宽）
    And 但后端保存的"最终导出图"需满足：
    - 分辨率：至少 300 DPI 对应的像素尺寸（假设纪念钞物理尺寸 160mm × 80mm，则需要至少 1890 × 945 像素）
    - 格式：PNG（无损压缩）
    - 色彩空间：sRGB
  # 如果生图模型的原生输出分辨率不足，需要在生成后通过超分辨率算法（如 ESRGAN）放大。
```

### 13.3 书签设计约束

```
Scenario 10.4 — 书签的设计约束（与纪念钞的主要差异）
  Given 用户在快速模式或精细模式下设计书签
   When AI 大脑组装书签的生图提示词
   Then 提示词模板与纪念钞有以下关键差异：

    (1) 无面值、无编号
    - 书签不是"钞票"，不需要面值和序列号
    - 模板中不包含 denomination、serial number 等关键词

    (2) 版面比例
    - 默认竖版窄长形（宽高比约 1:3），如 50mm × 150mm
    - 提示词中强调 "narrow tall vertical bookmark design"

    (3) 典型元素
    - 主角人物通常在书签上半部或中部
    - 底部可留白或放置励志语句/角色名言
    - 顶部可能有打孔区域（圆孔）或丝带/流苏效果
    - 边框通常是简约装饰线而非复杂的扭索纹

    (4) 精细模式下的设计分区差异
    - 移除：面值分区、编号规则分区、发行机构分区
    - 保留：整体风格、配色方案、主角元素、装饰元素
    - 新增：书签文字分区（替换面值区，用于输入角色名言或个性化文字）
    - 新增：顶部样式（打孔/丝带/无）


Scenario 10.5 — 书签编辑器 UI 差异
  Given 用户正在编辑一个书签类型的项目
   Then 画布的宽高比为 1:3（竖版窄长形）
    And 编辑页没有"正面/反面"切换 Tab
    And 精细模式的设计分区面板不包含面值和编号相关分区
    And 快速模式的描述框 placeholder 不同：
    "描述你想要的书签，例如：鸣人站在木叶村剪影前，竖版构图，底部配上'这就是我的忍道'"
```

### 13.4 海报设计约束

```
Scenario 10.6 — 海报的设计约束（与纪念钞/书签的差异）
  Given 用户在快速模式或精细模式下设计海报
   When AI 大脑组装海报的生图提示词
   Then 提示词模板的关键特征：

    (1) 无面值、无编号
    - 海报是展示性产品，不需要钞票元素

    (2) 版面比例
    - 默认接近 A4/A3 比例（约 1:1.414）
    - 提示词中强调 "poster design, large format, display piece"

    (3) 典型元素
    - 大面积的视觉冲击（主角占据画面主要部分）
    - 标题文字（如动漫名称、角色名）以大字体显示
    - 可包含上映日期、Slogan 等海报常见元素
    - 边框可选（全出血无边框 / 白色边框 / 装饰边框）

    (4) 精细模式下的设计分区差异
    - 移除：面值分区、编号规则分区、防伪底纹分区
    - 保留：整体风格、配色方案、主角元素、文字内容、装饰元素
    - 新增：标题文字分区（大标题 + 副标题）
    - 新增：排版布局选择（居中对称 / 左右分栏 / 上下分割 / 散点式）


Scenario 10.7 — 海报编辑器 UI 差异
  Given 用户正在编辑一个海报类型的项目
   Then 画布的宽高比接近 A4（约 1:1.414）
    And 编辑页没有"正面/反面"切换 Tab（单面）
    And 精细模式的设计分区面板包含"标题文字"和"排版布局"分区
    And 快速模式的描述框 placeholder 不同：
    "描述你想要的海报，例如：赛博朋克风格，主角在画面中央偏右，上方大标题'AKIRA'，霓虹配色"
```

### 13.5 物理印刷规格（待确认，仅纪念钞）

```
Scenario 10.8 — 纪念钞物理印刷规格（待与印刷厂确认）
  待确认项目：
  - 纪念钞的物理尺寸（建议：160mm × 80mm，与真实钞票的常见尺寸相近）
  - 纸张材质（铜版纸？特种防伪纸？）
  - 是否覆膜？
  - 是否烫金/烫银？
  - 印刷精度（DPI）
  - 出血位要求（设计图是否需要在裁切线外多预留 3mm？）
  - 正反面图像的套准要求
  - 最小字体大小（过小的文字可能印不出来）
```

### 13.6 产品类型对社区功能的影响

```
Scenario 10.9 — 社区排行榜中展示不同产品类型
  Given 系统中有各种类型的公开项目
   When 用户在社区排行榜中浏览
   Then 每个项目卡片上显示产品类型图标（🪙/📑/🖼️）
    And 排行榜可按产品类型筛选
    And 纪念钞项目展示正反面切换预览（缩略图旁有"1/2"翻页图标）
    And 书签和海报项目仅展示单面预览

Scenario 10.10 — 非纪念钞项目也可以公开和 Fork
  Given 用户有一个书签或海报项目，状态为"已完成"
   When 用户点击"公开到社区"
   Then 行为与纪念钞公开一致（确认 → 公开到排行榜）
    And 其他用户可以 Fork 书签/海报项目（复制设计，在自己名下继续编辑或导出）
    And 书签/海报的 Fork 复用相同的溯源链逻辑
  # 设计意图：社区不区分产品类型。好设计无论什么品类都值得被看到和复用。
```
