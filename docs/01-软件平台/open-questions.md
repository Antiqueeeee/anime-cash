# 待确认问题清单

> 最后更新：2026-07-20
>
> 本文档追踪项目中尚未敲定的决策事项。每个事项标注了优先级、影响范围、以及当前的最佳猜测（如果已有倾向的话）。

---

## 高优先级（影响核心流程，需要尽快决策）

### Q1: 激活码售卖方式

**背景**：用户需要在电商平台购买激活码，然后在平台内兑换。

**待决定**：
- 管理员在后台手动生成一批激活码 → 手动上架到淘宝/微信小店？（简单可靠，适合起步）
- 还是对接电商 API 自动化？（如淘宝开放平台/有赞 API）
- 如果是手动：激活码的格式是什么？一批生成多少个？

**影响范围**：Feature 8（激活码）、后台管理系统设计、运营流程

**倾向**：起步阶段手动生成+手动上架，验证商业模式后再考虑自动化。

---

### Q2: AI 大脑技术方案

**背景**：AI 大脑负责理解用户意图并组装生图提示词，是系统的核心技术组件。

**待决定**：
- 自研 LLM agent（如 GPT-4 / Claude 的 function calling + 自己写的 prompt template + 后端编排逻辑）
- 还是接入现有 agent 框架（如 pi-agent、LangChain、CrewAI 等）
- 如果自研：使用哪个 LLM？（GPT-4o / Claude Sonnet / 国内模型如 DeepSeek、通义千问）

**影响范围**：Feature 6（AI 大脑），几乎是整个系统的核心技术决策。决定了系统对用户描述的理解质量。

**倾向**：待讨论。自研方案灵活性高、成本可控；接入框架速度更快。

---

### Q3: 生图模型选择

**背景**：实际生成纪念钞/书签/海报预览图的 AI 模型。

**待决定**：
- Stable Diffusion（自部署）：成本低，可无限调用，但需要 GPU 服务器运维，图像质量依赖模型和 LoRA
- Midjourney API：效果好但贵，API 可能有调用限制
- DALL·E 3：效果好，价格中等
- 国内模型（通义万相、文心一格、CogView 等）：合规性好，中文理解可能更优
- 是否需要支持多个模型让用户选择？（如"写实风格用 SD，二次元风格用 NAI"）

**影响范围**：Feature 4、5、6（所有设计功能），直接决定生成效果和平台运营成本

**倾向**：待讨论。可能需要根据产品类型（纪念钞/书签/海报）使用不同的模型或 LoRA。

---

### Q4: 物理印刷规格

**背景**：纪念钞最终要印刷为实体产品。

**待决定**：
- 纪念钞的精确物理尺寸？（建议 160mm × 80mm）
- 纸张材质？（铜版纸 / 特种防伪纸 / 其他）
- 是否覆膜？是否烫金/烫银？
- 印刷精度（DPI）？推荐 ≥ 300 DPI
- 出血位要求？（设计图是否需要在裁切线外多预留 3mm？）
- 最小字体大小限制？
- 印刷厂的起订量？单张成本？

**影响范围**：Feature 7（提交与订单）、Feature 10（设计约束）、后台印刷流程、定价

**倾向**：需要先联系印刷厂确认后再定。

---

### Q5: 新产品类型的设计约束

**背景**：用户要求项目类型不限于纪念钞，还有书签、海报等。不同产品类型有不同的设计约束。

**待决定**：
- 除了纪念钞、书签、海报，还有哪些产品类型？
- 书签的物理尺寸和设计约束？（如 50mm × 150mm 竖版）
- 海报的物理尺寸和设计约束？（如 A3 / A4，是否支持自定义尺寸？）
- 非纪念钞类型的产品，设计完成后如何交付给用户？
  - 仅支持导出高清图片下载？
  - 还是也会考虑后期接入印刷？（目前用户只说纪念钞能打印）
- 每种产品类型的正面/反面概念是否相同？（书签可能只有单面？海报可能不需要"反面"？）

**影响范围**：Feature 2（项目管理）、Feature 4-5（设计模式）、Feature 10（设计约束）、Feature 7（提交与订单）

**倾向**：待讨论。首期可以先做 3 种类型（纪念钞/书签/海报），每种有独立的提示词模板和设计约束，但共享同一套设计工具。

## 中优先级（影响用户体验，但可以用默认方案先行）

### Q6: 排行榜算法细节

**当前默认方案**（见 BDD Scenario 9.5）：
- 热度 = 点赞数 × 3 + Fork 数 × 5 + 浏览量 × 1
- 时间衰减：半衰期 30 天，衰减因子 = 1 / (1 + days/30)

**待确认**：
- 是否需要区分"热门"和"趋势"两个榜单？
- 时间衰减参数是否需要调整？
- 是否按产品类型（纪念钞/书签/海报）分榜？

**影响范围**：Feature 9（社区排行榜）

---

### Q7: 素材网络搜索来源

**待决定**：
- 使用哪个图片搜索 API？
  - Unsplash API（免费额度，素材偏摄影，不太适合动漫）
  - Bing Image Search API（付费，覆盖面广）
  - 自建爬虫（搜 Pixiv/DeviantArt 等，但有版权和法律风险）
- 版权免责声明的法律有效性？

**影响范围**：Feature 3（素材管理）

**倾向**：首期使用 Bing Image Search，附带版权提示。或直接先不支持网络搜索，只做"本地上传 + 常用素材"两 Tab。

---

### Q8: "设计草稿保存为模板"功能

**背景**：精细模式下，用户可能花了很多时间调参数。能否将当前配置保存为模板，下次新建项目时一键套用？

**待决定**：是否需要这个功能？首期还是后期？

**影响范围**：Feature 5（精细模式）

**倾向**：首期不做的概率较大（避免 scope creep）。用户可以通过 Fork 自己之前的项目实现类似效果。

---

## 低优先级（首期可不做，但值得记录）

### Q9: 多语言支持

- 首期是否只需要中文？
- 后续是否需要日文（二次元用户群体）？英文？
- 国际化对 UI 设计的影响（如中英文文本长度差异）

**倾向**：首期仅中文。

---

### Q10: Fork 时是否给原创作者分润

- 如果用户的 Fork 或直接打印为平台带来收入，是否分一部分给原创作者？
- 这是一个商业模式问题，关系到社区激励机制

**倾向**：首期不做。验证社区活跃度后再考虑。

---

### Q11: 是否需要"多人协作设计"

- 多个用户共同编辑一个项目？
- 类似于 Figma 的多人协作 vs 更简单的"只读分享链接"

**倾向**：首期不做。先做好单人设计体验。

---

### Q12: 纪念钞是否可以有多于两面

- 目前假设只有正面和反面
- 未来是否需要折叠式纪念钞（如三折页）？

**倾向**：首期仅支持正反两面。
